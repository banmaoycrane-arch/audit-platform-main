Tare 会计业务开发项目规则
一、核心设计原则
可读性优先：牺牲少量简洁性换取语义清晰，禁止使用晦涩语法与过度简写，保证具备基础编程能力的会计人员可读懂
业务逻辑透明：所有财税计算逻辑必须显性化，会计口径、政策依据必须通过注释明确标注，满足审计可追溯要求
低复杂度约束：优先使用 Python 基础语法实现功能，禁用高阶函数、复杂装饰器、元编程等不易理解的写法；单函数代码不超过 80 行
数据零误差：所有金额、税额计算严格符合会计核算精度要求与科目勾稽关系
二、命名规范
统一采用 snake_case（小写 + 下划线）命名风格，全程使用完整英文单词，禁止无意义缩写。
变量命名：必须携带业务口径与单位属性，明确标注含税 / 不含税、借方 / 贷方、本期 / 累计等口径
正例：sales_amount_excl_tax（不含税销售额）、vat_output_payable（销项应纳税额）
反例：amt、tax、data
函数命名：以动词开头，格式为「动作_业务对象」，明确表达业务功能
正例：calculate_monthly_vat_payable（计算月度增值税应纳税额）、generate_accounting_voucher（生成记账凭证）
反例：calc_vat、gen_vou
文件与目录命名：全小写 + 下划线，按业务模块划分，文件名直接体现用途
正例：vat_calculation.py（增值税计算）、bank_reconciliation.py（银行对账）
类命名：采用大驼峰（PascalCase），以业务实体命名
正例：VatDeclarationForm（增值税申报表）、AccountingVoucher（记账凭证）
三、注释规范（强制执行）
所有代码必须配备充足注释，以「会计人员能看懂业务逻辑」为合格标准。

逻辑块与行内注释
每个独立业务步骤前必须加注释，说明该步骤的业务目的
会计判断节点、特殊调整项必须单独标注原因与政策依据
示例：
【财税〔2016〕36 号】经纪代理服务差额计税，代收款项不计入销售额
taxable_sales = total_received - agent_payment
四、代码格式规范
遵循 PEP8 简化版，降低学习与维护成本
缩进：统一使用 4 个空格，禁止使用制表符（Tab）
行宽：单行代码不超过 100 个字符，超长表达式换行对齐
空行规则：不同业务逻辑块之间空 1 行；两个函数之间空 2 行；文件末尾保留 1 个空行
导入规范：按「标准库 → 第三方库 → 本地模块」顺序分组，组之间空 1 行；禁止使用通配符导入
表达式：运算符前后加空格，逗号后加空格
五、会计业务代码强制规则
金额精度强制要求
所有金额计算禁止使用 float 浮点型，统一使用 decimal.Decimal 类型，避免二进制浮点误差
金额统一保留 2 位小数，四舍五入规则与税务、财务系统保持一致
标准写法示例：
from decimal import Decimal, ROUND_HALF_UP
tax_amount = (amount * tax_rate).quantize (Decimal ("0.00"), rounding=ROUND_HALF_UP)
业务参数配置化
税率、科目代码、征收率、扣除标准等可能变动的参数，统一存放于 config 目录的配置文件，禁止散落在业务代码中
政策变更时仅需修改配置文件，无需改动业务逻辑代码
计算过程可追溯
核心财税计算必须保留中间结果，支持分步对账，禁止一步写出最终结果
关键计算节点输出日志，记录每一步的数值，满足审计留痕要求
勾稽校验强制化
所有计算结果必须配套校验逻辑，利用会计平衡关系自动排查错误
必做校验项：价税合计校验、记账凭证借贷平衡校验、财务报表勾稽校验
校验不通过时必须抛出带业务语义的异常，禁止静默忽略

七、错误处理与日志规则
异常报错必须携带业务语义，禁止仅输出纯技术错误描述
正例：raise ValueError ("销项税额计算异常：不含税销售额为负数，请检查发票源数据")
反例：raise ValueError ("data error")
关键业务节点必须记录日志，包含时间、业务动作、计算结果
所有输入数据必须做合法性校验（金额非负、日期格式正确、税率在合理范围等）
八、Tare 多 Agent 协同执行规则
代码生成 Agent：严格遵循本规则，优先保证注释完整、逻辑清晰，禁止使用炫技式写法
代码审核 Agent：重点核查三项内容：会计口径准确性、金额精度合规性、注释完整易懂性
测试 Agent：必须使用标准会计案例作为测试用例，验证计算结果与手工核算结果的一致性，金额误差不得超过 0.01 元
