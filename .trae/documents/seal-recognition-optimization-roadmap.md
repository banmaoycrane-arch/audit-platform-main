# 印章识别优化项目路线图

## 一、项目背景与目标

### 1.1 背景

当前系统在处理合同等财务文档时，已集成多模态大语言模型（LLM）进行文字内容与语义识别。然而，对于合同中仅以印章形式存在、未在正文中重复说明的主体信息（如合同专用章、财务章、法人章等），现有方案存在识别率低、关联主体失败的问题。为提升合同审计证据的完整性与合规性，需引入专用印章检测与识别能力，并逐步构建"检测 → 识别 → 校验 → 知识库匹配"的混合处理管线。

### 1.2 总体目标

- **短期**：基于现有 OCR 框架（EasyOCR / PaddleOCR），实现印章区域的定位、提取与文字识别，形成可落地的 MVP。
- **中期**：引入多模态 LLM 对印章检测结果进行关联校验，提升主体识别准确率。
- **长期**：建立规则引擎与印章知识库，实现印章合法性、一致性、合规性的自动核验。

### 1.3 文档用途

本路线图为项目团队、前后端开发人员、测试人员及审计业务方提供清晰的阶段划分、资源投入、里程碑节点与验收标准，确保印章识别优化工作可控、可追踪、可验收。

---

## 二、项目实施原则

1. **渐进式演进**：从低成本、高可行性的 OCR 增强方案起步，逐步叠加 AI 校验与规则引擎。
2. **可审计性**：所有识别结果必须保留原始坐标、置信度、处理阶段与人工复核入口。
3. **业务优先**：技术方案需服务于审计内部控制目标，印章识别结果必须能关联到合同主体、签署日期、签署地点等关键要素。
4. **前后端协同**：后端负责识别算法与数据模型，前端负责证据展示、结果叠加与人机交互。
5. **不破坏现有流程**：新增印章识别模块不得影响已有合同解析、校验、合规评分主流程。

## 四、第一阶段：印章检测与 OCR 识别（MVP）

### 4.1 阶段范围

#### In Scope（本阶段必须完成）

1. **印章区域检测模块**
   - 支持输入格式：PNG、JPG、PDF（需先转换为图片）。
   - 检测目标：圆形章、椭圆形章、矩形章等常见印章外形。
   - 输出：每个印章的边界框（bounding box，像素坐标）、置信度、印章类型初步判断。

2. **印章区域提取模块**
   - 根据边界框裁剪印章区域。
   - 执行颜色分离（红色/蓝色通道增强）、二值化、去噪、轻微透视校正，提升 OCR 识别率。
   - 输出：标准化后的印章子图。

3. **印章文字识别模块**
   - 基于现有 EasyOCR 或 PaddleOCR 框架进行印章内文字识别。
   - 输出：识别文字列表、每个文字的坐标位置、置信度。
   - 建立文字内容与坐标位置的关联记录，支持按环形/弧形文字进行排序与重组。

4. **结果存储与数据模型**
   - 新建 `ContractSeal` 数据模型，至少包含：
     - `id`、`contract_id`、`source_file_id`
     - `bbox`（边界框 JSON）
     - `seal_image_path`（印章子图存储路径）
     - `recognized_text`（识别到的原始文字）
     - `text_items`（文字 + 坐标数组）
     - `seal_type`（印章类型，初判）
     - `confidence`（整体置信度）
     - `created_at`、`updated_at`
   - 识别结果持久化到数据库，并支持按合同查询。

5. **后端 API**
   - `POST /api/v1/contracts/{contract_id}/seals/extract`：触发印章提取任务。
   - `GET /api/v1/contracts/{contract_id}/seals`：获取该合同下的印章识别结果列表。
   - `GET /api/v1/seals/{seal_id}`：获取单个印章详情。

6. **前端界面预留**
   - 在合同详情页（Contract Detail）新增"印章证据"展示区域。
   - UI 布局预留：左侧原图 + 印章高亮框，右侧识别结果列表。
   - 预留数据接口与渲染逻辑，支持后续印章结果动态展示与叠加。
   - 保证在印章数据为空时的兼容展示与用户体验。

#### Out of Scope（本阶段明确不做）

1. 不训练新的深度学习印章检测模型，仅使用传统 CV 或现有轻量方案。
2. 不实现多模态 LLM 印章关联校验（第二阶段）。
3. 不实现印章真伪鉴别、公章备案库联网核验（第三阶段）。
4. 不修改合同解析、合规评分等现有主流程，仅做结果关联。

### 4.2 技术方案

#### 4.2.1 印章区域检测

**方案 A：传统计算机视觉（推荐 MVP）**

- **颜色阈值分割**：红色印章在 HSV 空间中具有较稳定的色相范围，先通过颜色过滤提取候选区域。
- **形状检测**：使用轮廓检测（OpenCV `findContours`）+ 最小外接圆/椭圆拟合，筛选出符合印章几何特征的连通区域。
- **后处理**：过滤过小区域、非闭合轮廓、低饱和度噪声。

**方案 B：轻量深度学习模型（备选）**

- 若传统 CV 在复杂背景、重叠印章、低分辨率场景下效果不佳，可引入 PaddleOCR 的印章检测模型或 YOLO 轻量模型。
- 本阶段以方案 A 为主，方案 B 作为技术预研。

#### 4.2.2 印章文字识别

- 直接复用现有 OCR 服务：`EasyOCR` 或 `PaddleOCR`。
- 针对印章文字特点进行预处理：
  - 红色通道增强与背景漂白。
  - 极坐标变换：对环形文字进行拉直，转化为水平文本行。
  - 字符间距较大的印章文字，可适当进行膨胀操作以连接断裂笔画。
- 输出按坐标排序，环形章按角度顺时针/逆时针重组为完整文字串。

#### 4.2.3 坐标关联记录

- 每个识别字符保存 `text`、`x`、`y`、`width`、`height`、`confidence`。
- 整体印章保存 `bbox`：`{x1, y1, x2, y2}`，采用左上角坐标系。
- 坐标单位为像素，基于原始图片分辨率。

### 4.5 验收标准

#### 4.5.1 功能验收

- [ ] 能够成功上传合同文件并触发印章提取任务。
- [ ] 返回的每个印章包含边界框坐标、置信度、印章子图路径、识别文字、文字坐标数组。
- [ ] 印章数据可持久化，支持按合同 ID 查询。
- [ ] 合同详情页出现"印章证据"区域，布局合理，无数据时友好提示。

#### 4.5.2 性能验收

- [ ] 单页 A4 合同印章检测平均耗时 ≤ 3 秒（不含网络传输）。
- [ ] 单合同多印章（≤5 个）整体提取耗时 ≤ 10 秒。
- [ ] 并发处理 5 个合同时系统稳定，不出现内存溢出或服务挂起。

#### 4.5.3 准确率验收

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 印章检出率（Recall） | ≥ 85% | 真实印章中被正确检出的比例 |
| 印章误检率（False Positive Rate） | ≤ 15% | 非印章区域被误判为印章的比例 |
| 印章文字识别准确率 | ≥ 70% | 识别文字与人工标注一致的比例 |
| 坐标位置误差 | ≤ 10 像素 | 边界框与真实印章边缘的平均偏差 |

> 说明：以上指标基于 50 份测试样本统计。MVP 阶段允许通过人工复核补充识别失败案例。

#### 4.5.4 代码与工程验收

- [ ] 新增模块代码覆盖率 ≥ 80%。
- [ ] 所有新增 API 具备 Swagger/OpenAPI 文档。
- [ ] 关键识别步骤记录日志，支持审计追溯。
- [ ] 前端组件通过 TypeScript 类型检查，build 通过。

## 五、第二阶段：多模态 LLM 关联校验

### 5.1 阶段目标

在第一阶段检测到印章并提取文字后，引入多模态 LLM 对印章图像与合同上下文进行联合分析，实现：

1. 印章类型精确分类（公章、合同专用章、财务专用章、法人章、发票章等）。
2. 印章主体与合同甲方 / 乙方 / 丙方的自动关联。
3. 印章与正文签署主体不一致的异常标记。
4. 印章模糊、缺角、重叠等质量问题的识别与提示。

### 5.2 核心任务

1. **多模态提示词工程**
   - 输入：印章裁剪图 + 印章文字 + 合同主体列表 + 签署位置上下文。
   - 输出：JSON 格式，包含 `seal_type`、`associated_party`、`confidence`、`anomalies`。

2. **LLM 校验服务**
   - 封装为独立服务 `SealVerificationService`。
   - 支持批量校验、失败重试、结果缓存。

3. **结果融合**
   - 将 OCR 识别结果与 LLM 校验结果合并，生成最终印章记录。
   - 对冲突结果（如 OCR 识别主体与 LLM 关联主体不一致）标记为"需人工复核"。

### 5.3 验收标准

- [ ] 印章类型分类准确率 ≥ 85%。
- [ ] 印章主体关联准确率 ≥ 80%。
- [ ] 异常标记召回率 ≥ 75%。
- [ ] 校验结果以置信度分层，低置信度结果必须进入人工复核队列。

## 六、第三阶段：规则引擎与知识库匹配

### 6.1 阶段目标

构建印章合规性核验能力，将印章识别结果与审计内部控制规则、客户供应商主数据、工商注册信息进行匹配。

### 6.2 核心任务

1. **印章规则库**
   - 定义规则：
     - 合同金额超过阈值必须使用公章或合同专用章。
     - 财务章不得用于签署采购/销售合同。
     - 法人章通常需与公章配套使用。
     - 印章名称应与合同主体名称一致。

2. **知识库匹配**
   - 将印章中的企业名称、统一社会信用代码与系统中客户 / 供应商 / 往来单位主数据匹配。
   - 支持人工维护印章白名单 / 黑名单。

3. **合规评分集成**
   - 将印章合规结果接入 `contract_validation_service.py` 的采购合同合规性评分。
   - 新增"印章合规"评分维度，权重建议 5%-10%。

### 6.3 验收标准

- [ ] 规则引擎支持至少 10 条印章合规规则的配置化扩展。
- [ ] 印章主体与主数据匹配准确率 ≥ 90%（在数据已维护前提下）。
- [ ] 印章合规评分结果可解释，每条扣分项附带原因与原文证据。
- [ ] 支持审计底稿导出，包含印章图片、识别结果、校验意见。

## 七、前端开发要求

### 7.1 合同详情页印章证据区域

#### 7.1.1 位置与布局

- 位置：合同详情页签（Tabs）新增"印章证据"标签，位于"合同内容"与"解析结果"之后。
- 布局：
  - 左侧 60%：原始合同图片 / PDF 预览，叠加印章边界框高亮。
  - 右侧 40%：印章列表，每个印章展示缩略图、识别文字、置信度、印章类型、关联主体、操作按钮。

#### 7.1.2 交互设计

- 点击右侧印章项，左侧自动滚动并高亮对应印章。
- 支持放大查看印章子图。
- 每个印章提供"确认正确"、"标记异常"、"人工修正"入口（第一阶段可仅做 UI 预留，后续阶段实现功能）。
- 无印章数据时显示："未检测到印章，请确认文件清晰度或联系管理员。"

#### 7.1.3 数据接口预留

前端需预留以下接口调用逻辑：

```typescript
// 获取合同印章列表
GET /api/v1/contracts/{contract_id}/seals

// 触发印章提取
POST /api/v1/contracts/{contract_id}/seals/extract

// 获取印章详情
GET /api/v1/seals/{seal_id}
```

返回数据结构预留：

```typescript
interface SealTextItem {
  text: string;
  x: number;
  y: number;
  width: number;
  height: number;
  confidence: number;
}

interface ContractSeal {
  id: string;
  contract_id: string;
  bbox: { x1: number; y1: number; x2: number; y2: number };
  seal_image_url: string;
  recognized_text: string;
  text_items: SealTextItem[];
  seal_type: string | null;
  associated_party: string | null;
  confidence: number;
  anomalies: string[];
  created_at: string;
}
```

### 7.2 兼容性要求

- 支持主流浏览器：Chrome、Edge、Firefox。
- 响应式适配：最小宽度 1280px，平板端隐藏左侧预览或改为上下布局。
- 图片渲染使用 `canvas` 或 `SVG` 叠加边界框，确保缩放时坐标同步。

## 八、数据安全与审计要求

1. 印章图片与识别结果按合同级进行权限隔离，仅授权用户可访问。
2. 所有印章提取操作记录审计日志：操作人、时间、合同 ID、识别结果摘要。
3. 识别结果不作为最终审计结论，仅作为审计证据提示，需人工复核确认。
4. 印章图片存储路径不暴露原始服务器路径，通过签名 URL 或受控接口访问。

## 九、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 传统 CV 在复杂背景印章上检出率低 | 第一阶段准确率不达标 | 引入 PaddleOCR 印章检测模型作为备选方案 |
| 印章文字弧形排列导致 OCR 识别率低 | 文字识别准确率不达标 | 增加极坐标拉直预处理，或采用专用印章 OCR 模型 |
| PDF 扫描件分辨率差异大 | 检测与识别不稳定 | 统一在预处理阶段进行 DPI 标准化与超分增强 |
| 多印章重叠或盖章位置跨页 | 漏检或坐标错误 | 支持分页处理，增加重叠区域 NMS 后处理 |
| 前端 PDF 渲染与坐标叠加不同步 | 用户体验差 | 使用 pdf.js 等库统一坐标系，缩放时动态重算 bbox |
| 新增模块影响现有合同解析性能 | 系统稳定性下降 | 印章提取采用异步任务队列，不与主流程阻塞 |

## 十一、附录

### 11.1 术语表

- **印章检出率（Recall）**：真实印章中被正确检测出来的比例。
- **印章误检率（FPR）**：非印章区域被误判为印章的比例。
- **边界框（Bounding Box）**：包围印章区域的最小矩形，以像素坐标表示。
- **极坐标变换**：将环形文字投影为水平文字行的图像处理技术。
- **NMS（Non-Maximum Suppression）**：非极大值抑制，用于合并重叠检测框。
