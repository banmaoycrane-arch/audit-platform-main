> 版本：v0.2（边界修正）  
> 日期：2026-06-21  
> 状态：边界已冻结（合同 / 往来款项 / 银行资金 三分离）  
> 前置：`core-business-concepts-boundary.md`、现有 `register_ingestion_service` / `draft_archive_service`

## 0. 台账边界修正（v0.2 核心变更）

此前将「合同」登记到「往来账款台账」，并与银行流水共用 counterparty 语义维度，导致三类业务事实边界模糊。现冻结为 **三类 Register 互不重叠**：

```text
┌─────────────────┐     ┌─────────────────────┐     ┌─────────────────────┐
│  合同台账        │     │  往来款项台账        │     │  银行资金台账        │
│ contract_register│     │ counterparty_ledger │     │ bank_cash_flow      │
├─────────────────┤     ├─────────────────────┤     ├─────────────────────┤
│ 承诺 / 约定      │     │ 已发生的债权债务余额  │     │ 已发生的资金收付事实  │
│ 待执行·未执行    │     │ 应收/应付/预收/预付   │     │ 银行回单·对账单明细 │
│ 取消·变更        │     │ 对账单·函证余额      │     │ 未达账项·调节表      │
│ 不一定形成款项   │     │ 必然存在款项事实      │     │ 不等于往来余额       │
└────────┬────────┘     └──────────┬──────────┘     └──────────┬──────────┘
         │ 执行触发                  │ 发票/结算确认              │ 收付结算
         └──────────────────────────┴──────────────────────────┘
                              勾稽链（非重复登记）
```

| 问题 | 旧口径（已废弃） | 新口径（v0.2） |
|------|------------------|----------------|
| 合同底稿登记到哪？ | `counterparty_ledger` | **`contract_register`** |
| 银行流水是否同时登记往来？ | `bank_cash` → 银行 + 往来 | **仅 `bank_cash_flow`** |
| 往来台账登记什么？ | 合同、结算、银行混用 | **仅已发生余额事实**（发票未结清、对账单、函证） |
| 采购/销售模块定位 | 含「合同台账」字样 | **业务循环子集**，合同执行后的方向性明细 |

**链路关系（不重复落库）**：

```text
合同(承诺) ──执行──▶ 业务单据(入库/发票) ──确认──▶ 往来余额 ──收付──▶ 银行流水
   │                      │                        │              │
contract_register    purchase/sales           counterparty_ledger  bank_cash_flow
```

## 1. 规划目标

本规划回答三个问题：

1. **底稿分解后，各模块台账在审计业务里分别承担什么角色？**
2. **台账之间的细分关系，如何决定管理颗粒度与工作流？**
3. **底稿作为审计证据，如何独立管理版本、索引与更新？**

核心原则（与边界文档一致）：

- **Register（业务台账）≠ Ledger（会计账套）≠ AccountingEntry（会计分录）**
- 底稿（SourceFile）是证据载体；台账是业务事实登记；分录是会计确认结果
- 审计工作流在 **Project** 边界内编排，正式数据以 **ledger_id** 过滤

---

## 2. 三层架构总览

```text
┌─────────────────────────────────────────────────────────────────┐
│  L1  底稿证据层（Workpaper / SourceFile）                        │
│      上传 · 版本 · 索引 · 归档 · 函证附件 · 更新留痕              │
├─────────────────────────────────────────────────────────────────┤
│  L2  业务台账层（Register）                                      │
│      按审计业务流程拆分的子数据集 + 勾稽关系 + 差异项              │
├─────────────────────────────────────────────────────────────────┤
│  L3  会计确认层（AccountingEntry + COA）                         │
│      科目余额 · 发生额 · 截止性 · 列报 · 与总账对账               │
└─────────────────────────────────────────────────────────────────┘
         ▲                    ▲                    ▲
         │                    │                    │
    语义分解+归档        审计程序引擎          凭证生成/复核
```

**管理颗粒度由 L2 的拆分方式决定**：拆得越细，工作流节点越多、可审计性越强，但维护成本也越高。

## 3. 模块台账的最终定位（按审计业务流程）

### 3.1 资金收付台账（bank_cash_flow）

**本质**：真实资金对账域，不是简单的「银行流水列表」。

| 维度 | 定义 |
|------|------|
| 业务含义 | 银行/现金实收实付的**发生事实**登记 |
| 审计程序 | 银行函证 → 重新计算调节 → 确认与计量 → 与总账货币资金科目对账 |
| 子数据集 | 银行账户主档、银行对账单明细、未达账项、调节表、函证回函 |
| 勾稽对象 | `AccountingEntry` 中 1001/1002 等科目；`Counterparty` 中的银行/关联方 |
| 输出 | 银行调节表、函证控制表、资金收支异常清单 |

**工作流（审计系统下独立子流程）**：

```text
底稿上传(对账单/回单)
  → 解析登记 bank_cash_flow 台账
  → 与科目余额比对（调节表草稿）
  → 发起/登记函证（confirmation）
  → 回函底稿版本挂接
  → 重新计算（recompute）调节项
  → 审计结论：确认 / 保留意见 / 需追加程序
```

**与现有实现差距**：

- 现有：`BankStatement` 表 + 内存 `module_ledgers["bank_cash_flow"]` + 占位前端
- 缺：函证实体、调节表引擎、与 GL 对账 API、审计程序状态机

---

### 3.2 合同台账（contract_register）

**本质**：**承诺与约定**登记域，记录合同生命周期，不假定已形成会计意义上的往来余额。

| 维度 | 定义 |
|------|------|
| 业务含义 | 签约义务、权利范围、交付/付款条款 |
| 状态机 | `pending` 待执行 → `executing` 执行中 → `completed` 已完成；`not_executed` 未执行；`cancelled` 已取消 |
| 典型底稿 | 采购/销售/服务合同、框架协议、订单 |
| 审计关注点 | 合同真实性、条款完整性、截止日前后签约、承诺与执行一致性 |
| **不登记** | 应收应付余额、银行收付事实 |

采购/销售模块从合同**派生业务方向**（买/卖），但不替代合同台账本身。

---

### 3.3 往来款项台账（counterparty_ledger）

**本质**：**已发生的债权债务余额**登记域——款项必然存在，仅方向（采购/销售）或科目细分待勾稽。

| 维度 | 定义 |
|------|------|
| 业务含义 | Counterparty × 余额方向（应收/应付/预收/预付/其他） |
| 典型底稿 | 增值税发票（未结清）、往来对账单、余额函证回函、结算单 |
| 审计程序 | 函证 → 账龄 → 与 GL 科目对账 → 与银行收付勾稽 |
| **不登记** | 纯合同承诺、银行流水明细（流水在 bank_cash_flow） |

与采购/销售的关系：

```text
         会计科目（1122/2202/2203/1123 …）
                    │
     ┌──────────────┼──────────────┐
     ▼              ▼              ▼
往来款项台账    采购业务台账    销售业务台账
(余额事实)     (采购循环明细)   (销售循环明细)
     └──── 内部审计：同一 counterparty + period 三角勾稽 ────┘
```

- **采购/销售**：业务循环视角（合同执行→单证→余额变动），不是「合同台账」
- **内部审计**：采购付款合计 ↔ 应付变动 ↔ 银行付款（三处数据勾稽，非三处重复登记）

---

### 3.4 资金收付台账（bank_cash_flow）— 与往来款项的边界

**银行流水 ≠ 往来余额**。付款行为在银行台账记**资金事实**；对应应付减少在往来台账勾稽，不重复写入。

| 银行资金台账 | 往来款项台账 |
|--------------|--------------|
| 银行账户实收实付 | 对某单位的应收/应付余额 |
| 调节表、未达账项 | 函证、账龄、对账单 |
| 与 1001/1002 对账 | 与 1122/2202 等对账 |
| 底稿：对账单、回单 | 底稿：余额函证、结算单 |

**工作流**：

```text
底稿(对账单) → bank_cash_flow
  → 与 GL 货币资金调节
  → 与 counterparty_ledger 勾稽「本期收付是否解释余额变动」
  → 差异进入审计发现（不自动改账）
```

---

### 3.5 税务发票台账（tax_invoice）

**本质**：纳税义务与抵扣链条的登记，服务于「发生额核对 + 截止性 + 分类」。

- 勾稽：销项/进项 ↔ 收入/成本科目 ↔ 申报表
- 与往来款项：发票确认未结清余额时可联动 `counterparty_ledger`（余额事实，非重复）
- 与资金台账：大额票款核对通过**勾稽**完成，不在银行台账重复登记往来

---

### 3.6 其他模块（库存、薪酬、固定资产）

按业务循环挂载，优先级次于合同/往来/资金三角。  
每模块遵循同一模式：**Register 子数据集 + 与 GL 科目映射 + 审计程序模板**。

## 4. 台账细分关系 → 管理颗粒度矩阵

| 颗粒度 | 适用场景 | 台账拆分 | 工作流复杂度 |
|--------|----------|----------|--------------|
| **粗** | 代理记账、简单尽调 | 仅 SourceFile 归档 + 总账 | 低 |
| **中** | 常规年报审计 | 模块台账 + 按单位汇总往来 | 中 |
| **细** | 上市审计、IPO、专项资金审计 | 模块子数据集 + 函证 + 调节 + 三单匹配 + 版本底稿 | 高 |

**规则**：

- Project `type=audit` 默认启用「中」；可配置升级到「细」
- `type=accounting` 默认「粗～中」，不强制函证流
- 颗粒度在 **Project 设置** 中冻结，影响 UI 显示的模块深度与必填审计程序

## 5. 底稿管理模块（审计系统下独立域）

### 5.1 为什么需要独立模块

当前 `SourceFile` + `notes.archive` 仅解决「归位检索」，不满足：

- 底稿**版本**（修订、补充、替换）
- 底稿**索引**（索引号、交叉引用、工作底稿目录）
- 底稿**更新留痕**（谁、何时、为何更新）
- 底稿与审计程序、函证、结论的**挂接**

因此需要在**审计工作台（Audit Workspace）**下设立 **「工作底稿」** 一级模块，与「审计项目六步」并列。

### 5.2 建议数据模型（规划）

```text
WorkpaperIndex          # 底稿索引（项目级目录）
  ├─ index_no           # 如 A1.2.3
  ├─ project_id
  ├─ ledger_id (可选)
  ├─ audit_area         # 货币资金 / 收入 / 采购 ...
  └─ parent_id          # 树形结构

WorkpaperVersion        # 底稿版本
  ├─ source_file_id     # 关联物理文件
  ├─ version_no         # 1, 2, 3 ...
  ├─ status             # draft / submitted / reviewed / superseded
  ├─ prepared_by / reviewed_by
  ├─ change_reason
  └─ supersedes_id      # 上一版本

WorkpaperLink           # 底稿勾稽
  ├─ from_version_id
  ├─ to_type            # register_entry / finding / confirmation / entry
  ├─ to_id
  └─ link_type          # supports / contradicts / calculates
```

**与现有 SourceFile 关系**：

- `SourceFile` 继续作为**文件存储载体**（第一版可 1:1）
- `WorkpaperVersion` 作为**审计语义上的底稿版本**
- `draft_archive_service` 的虚拟路径 → 迁移为 `WorkpaperIndex` 的目录节点

### 5.3 底稿生命周期

```text
上传 → 解析登记台账 → 生成索引号 → 归档到项目目录
  → [修订] 新版本（旧版 superseded，保留只读）
  → [复核] 状态 reviewed
  → [导出] 工作底稿包（含索引清单 + 版本清单）
```

### 5.4 前端模块规划（审计系统）

| 路由 | 模块 | 功能 |
|------|------|------|
| `/audit/workpapers` | 工作底稿库 | 按项目/索引号/审计领域浏览 |
| `/audit/workpapers/{id}` | 底稿详情 | 版本历史、预览、勾稽、评论 |
| `/audit/confirmations` | 函证管理 | 函证发起、回函挂接、差异 |
| `/audit/reconciliations` | 调节与对账 | 银行调节、往来函证调节 |
| `/audit/register-crosscheck` | 台账内部审计 | 采购↔往来↔银行 三角勾稽 |

## 6. 底稿分解引擎的演进方向

当前 `draft_semantic_decomposition_service` 做**多维语义分解**（一条合同 → 合同台账 + 采购/销售；银行流水仅资金台账；往来余额仅来自发票/对账类底稿）。

**分解路由禁令（v0.2）**：

- 合同 → `contract_register`（+ 可选 `purchase`/`sales`），**禁止** `counterparty_ledger`
- 银行流水 → **仅** `bank_cash_flow`，**禁止** `counterparty_ledger`
- 发票 → `tax_invoice`（+ 可选 `counterparty_ledger` 当形成未结清余额）

最终应演进为 **「分解 + 路由 + 程序推荐」** 三合一：

| 阶段 | 能力 |
|------|------|
| **现已具备** | 规则/LLM 语义分解 → 多模块 register；虚拟归档路径 |
| **Phase 2** | 分解结果写入持久化 Register 视图（带 `ledger_id`）；推荐审计程序 |
| **Phase 3** | 自动创建 WorkpaperIndex 节点；版本 1.0 挂接；交叉引用建议 |
| **Phase 4** | 与函证/调节工作流联动；差异驱动新版本底稿 |

**分解输出扩展字段（规划）**：

```json
{
  "module_targets": [...],
  "recommended_audit_procedures": ["bank_confirmation", "counterparty_confirmation"],
  "workpaper_index_hint": "A2.1 货币资金",
  "cross_ledger_links": [
    {"from": "purchase", "to": "counterparty_ledger", "check": "amount_match"}
  ]
}
```

## 7. 与会计分录的边界（重申）

```text
底稿 ──分解──▶ Register 台账 ──审计程序──▶ 审计结论
                    │
                    └──规则/人工确认──▶ AccountingEntry 草稿 ──复核──▶ 正式凭证
```

- **资金收付台账**：对账结论不自动改账，差异进入「审计调整分录」草稿
- **往来/采购/销售**：三单差异是审计发现，不是自动冲销
- 所有「确认与计量」必须经过 **人工复核节点**（对齐 `AgentDraftReview` 模式）
