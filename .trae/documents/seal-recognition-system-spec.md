## 1. 项目背景与总体目标

### 1.1 业务痛点分析

当前多模态大型语言模型(LLM)在印章识别领域存在以下技术局限性与业务痛点：

| 痛点类别 | 具体描述 | 业务影响 |
|---------|---------|---------|
| **精度不足** | LLM对印章这类特殊图像的细节识别能力有限，尤其是低质量扫描件、模糊印章、变形印章等场景 | 识别准确率低，无法满足审计证据验证要求 |
| **缺乏空间信息** | LLM输出通常仅包含文字内容，丢失了印章在文档中的空间位置、大小、形状等关键信息 | 无法进行多印章比对、位置合规性检查 |
| **依赖外部服务** | 多模态LLM调用成本高、响应慢，且依赖外部API服务 | 系统可用性受外部服务影响，运营成本高 |
| **无审计追溯** | LLM推理过程黑盒化，无法提供识别过程的完整审计轨迹 | 不符合审计可追溯要求，无法满足合规检查 |
| **规则缺失** | LLM输出缺乏业务规则约束，无法进行印章类型校验、格式合规性判断 | 无法自动识别异常印章，需人工逐一审核 |

### 1.2 总体目标

分三个阶段递进式建设印章智能识别与校验系统，实现"检测→识别→校验→知识库匹配"的完整处理管线：

#### 第一阶段：印章智能识别（MVP）
- **核心能力**：基于传统计算机视觉(CV)技术实现印章区域检测、提取与文字识别
- **业务价值**：替代人工肉眼查找，提高印章识别效率，建立印章基础数据库

#### 第二阶段：多模态LLM关联校验
- **核心能力**：引入多模态LLM对识别结果进行语义校验，实现OCR与LLM结果融合
- **业务价值**：提升识别准确率，实现印章信息与合同内容的智能关联校验

#### 第三阶段：规则引擎与知识库匹配
- **核心能力**：构建印章规则库与企业主数据匹配系统，实现合规性评分
- **业务价值**：自动化印章合规性审查，提升审计证据可靠性与工作效率

## 2. 实施原则

### 2.1 渐进式演进

采用迭代开发模式，确保系统功能逐步完善并可随时交付可用版本：
- MVP阶段优先实现核心检测与识别能力
- 后续阶段在不影响现有功能的前提下增量扩展
- 每个阶段交付物需通过验收标准后方可进入下一阶段

### 2.2 可审计性

所有识别结果与处理过程需保留完整记录，支持追溯与审计：
- 记录每次印章提取的完整处理日志
- 保存检测框坐标、识别文字、置信度等中间结果
- 记录操作人、操作时间、操作来源等审计信息

### 2.3 业务优先

功能设计以解决实际业务问题为导向，优先满足核心业务需求：
- 优先实现印章检测、提取、识别的基础能力
- 后续扩展围绕审计实务需求设计（合规校验、异常识别等）
- 避免过度技术化设计，确保业务人员可理解、可使用

### 2.4 前后端协同

建立高效的前后端协作机制，确保接口设计与功能实现的一致性：
- API接口设计需经过前后端评审确认
- 前端类型定义与后端Schema保持同步
- 接口变更需通知相关模块并同步更新

### 2.5 兼容性保障

确保新系统与现有业务流程无缝集成，不影响现有工作流的正常运行：
- 印章识别作为独立模块，不侵入现有合同管理流程
- API设计遵循RESTful规范，便于集成
- 支持增量接入，可逐步扩展使用范围

## 4. 第一阶段MVP详细规划

### 4.1 印章区域检测

#### 4.1.1 算法选型

采用基于HSV颜色空间的传统计算机视觉方案：
- **颜色空间转换**：BGR → HSV，便于红色/蓝色印章分割
- **颜色阈值**：红色(H: 0-15, 160-180)，蓝色(H: 90-130)
- **形态学处理**：椭圆结构元素闭运算，连接断裂边缘
- **轮廓提取**：RETR_EXTERNAL模式，仅提取外轮廓

#### 4.1.2 参数设置

| 参数 | 默认值 | 说明 |
|-----|-------|-----|
| `min_area` | 300 | 印章最小像素面积，过滤噪点 |
| `max_area_ratio` | 0.8 | 相对图片面积的最大比例，防止误检整页 |
| `kernel_size` | (7, 7) | 形态学处理的结构元素大小 |
| `morph_iterations` | 2 | 闭运算迭代次数 |

#### 4.1.3 检测精度要求

| 指标 | 目标值 | 测试方法 |
|-----|-------|---------|
| 红色圆形印章检出率 | ≥ 90% | 使用合成印章图片测试集 |
| 蓝色椭圆印章检出率 | ≥ 85% | 使用合成印章图片测试集 |
| 误检率 | ≤ 5% | 无印章空白页测试 |
| 检测速度 | ≤ 2秒/页 | 标准A4扫描件(300dpi) |

#### 4.1.4 Fallback机制

当前MVP阶段预留PaddleOCR检测接口标记：
- 通过`detection_method`参数区分检测方法
- 当传统CV检测结果置信度低于0.5时，记录标记便于后续切换
- 第二阶段实现PaddleOCR作为备选方案

### 4.2 印章区域提取与预处理

#### 4.2.1 处理流程

```
裁剪 → 去噪 → 增强 → 倾斜校正 → 尺寸归一化 → 保存
```

#### 4.2.2 核心算法

| 处理步骤 | 算法 | 参数设置 | 业务目的 |
|---------|-----|---------|---------|
| **边界裁剪** | 安全裁剪 | 边界校验，防止越界 | 提取印章候选区域 |
| **双边滤波** | BilateralFilter | d=5, sigmaColor=50, sigmaSpace=50 | 去噪同时保留边缘 |
| **CLAHE增强** | 自适应直方图均衡 | clipLimit=2.0, tileGridSize=(8,8) | 提升对比度，便于OCR识别 |
| **饱和度提升** | HSV通道调整 | +15 | 增强印章颜色鲜明度 |
| **倾斜校正** | 最小外接矩形 | 角度阈值<5° | 校正轻微倾斜 |
| **尺寸归一化** | 等比例缩放 | target_size=256 | 统一输入尺寸，便于展示 |

#### 4.2.3 输出格式

- **文件格式**：PNG（无损压缩，保留印章细节）
- **文件命名**：UUID格式（`{uuid}.png`），避免原文件名泄露
- **存储路径**：`backend/app/storage/seals/`

### 4.3 文字识别与坐标关联

#### 4.3.1 OCR引擎

复用项目现有EasyOCR阅读器，支持中文识别：
- **语言模型**：ch_sim（简体中文）
- **模型加载**：延迟初始化，首次调用时加载
- **异常处理**：初始化失败或识别异常时返回空结果

#### 4.3.2 坐标映射

印章子图识别后的文字坐标需要映射回原始页面：
- **偏移量计算**：检测框左上角坐标作为offset
- **坐标转换**：子图内坐标 + offset = 原始页面坐标
- **空间位置**：保留每个文字项的x、y、width、height

#### 4.3.3 环形文字排序

针对圆形印章的环形文字，采用角度排序算法：
- **计算中心**：所有文字框中心的几何平均值作为印章中心
- **角度排序**：按`atan2(y - center_y, x - center_x)`逆时针排序
- **文字重组**：按排序结果拼接为完整字符串

| 置信度 | Confidence | 识别结果的可信度，范围0-1 |
| 多模态LLM | Multimodal LLM | 支持文本、图像等多种输入模态的大语言模型 |
| 合规评分 | Compliance Score | 评估印章符合业务规则程度的分数 |
