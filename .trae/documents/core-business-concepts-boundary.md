# 核心业务概念边界

## 目的

本文件用于冻结当前系统最容易混淆的核心业务概念，避免后续 specs 在 Team、Ledger、Project、Register、Accounting Entity、Counterparty、EntryTag、DocumentTag、organization_id 等概念之间反复摇摆。

本文件是需求边界治理文档，不直接修改数据库或代码。后续新需求、bugfix、spec 编写和验收都应优先对照本文件。

## 总原则

1. 财务正式数据必须有明确账簿边界。
2. 协作权限和正式核算边界不能混用。
3. 项目是工作任务边界，不等同于账簿。
4. 会计主体是报表/核算主体，不必然等同于团队或客户。
5. AI、标签、向量只能辅助理解，不替代正式会计规则。
6. 旧字段可以兼容，但新需求必须使用清晰主边界。

## 核心概念定义

| 概念 | 中文口径 | 新方案对应概念 | 主用途 | 不应承担的职责 |
|---|---|---|---|---|
| User | 用户 | 使用者 | 登录、操作人、权限主体、审计追踪 | 不代表客户、不代表团队 |
| Team | 团队 / 事务所 / 企业组织 | 使用者协作组织 | 协作与权限范围，管理成员和账簿 | 不作为正式核算数据边界 |
| Ledger | 会计账簿 | 核算账簿 / 法律单据承上启下维度 | 正式核算数据边界，凭证、期间、报表、审计范围的主过滤口径 | 不代表项目、不代表所有法律主体 |
| Project | 项目 | 审计工作任务边界 | 审计、记账、税务等工作任务边界 | 不直接替代账簿、不作为凭证主归属 |
| Register | 业务台账 | 业务台账 | 某业务模块内的业务流水或管理台账 | 不等于会计账簿，不直接替代凭证 |
| Accounting Entity | 会计主体 / 报表主体 | Reporting Entity / 可切换口径主体 | 报表、合并、内部核算主体判断 | 不等于 Team，不一定等于 Ledger |
| Legal Entity | 法律主体 | Primary Legal / Sub Legal | 营业执照、合同主体、责任主体 | 不直接决定凭证期间和账簿权限 |
| Tax Entity | 纳税主体 | Legal Entity（纳税口径） | 税号、税务申报、发票抬头 | 不直接替代会计主体 |
| Management Entity | 管理主体 | Accounting Unit / 管理维度 | 内部管理、事业部、利润中心等 | 不直接替代正式会计主体 |
| Counterparty | 往来单位 / 交易对象 | 交易对象 | 客户、供应商、关联方、政府、个人等交易对象 | 不承担借贷方向，不等于科目 |
| SourceFile | 原始资料文件 | 原始资料文件 | 合同、发票、银行回单、序时簿、图片、PDF 等证据载体 | 不等于正式凭证 |
| ImportJob | 导入任务 | 导入任务 | 一次文件导入或解析处理过程 | 不代表账簿或项目本身 |
| AccountingEntry | 会计分录 | 会计分录 | 正式或草稿凭证分录行 | 不承载完整文件证据原文 |
| EntryTag | 分录语义标签 | 分录语义标签 | 辅助核算、项目、部门、产品、客户、语义检索 | 不替代正式科目和借贷规则 |
| DocumentTag | 原始资料语义标签 | 原始资料语义标签 | 文件证据分类、业务性质、AI 检索 | 不替代 EntryTag，不直接入账 |
| AccountingPeriod | 会计期间 | 会计期间（天然影响 Scope） | 期间起止、结账、损益结转、快照 | 不负责凭证业务类型判断 |
| organization_id | 历史/兼容字段 | 被执行对象背景 / 关联方识别辅助 | 早期导入或组织口径兼容 | 新需求不应把它作为正式财务主边界 |

## 层级关系

建议理解为：

```text
User
  └─ Team：协作和权限范围
       └─ Ledger：正式会计账簿 / 财务数据边界
            ├─ AccountingPeriod：会计期间
            ├─ AccountingEntry：凭证与分录
            ├─ SourceFile / ImportJob：原始资料与导入任务
            ├─ Counterparty：往来单位
            ├─ ChartOfAccounts：会计科目
            └─ Accounting Entity：会计主体 / 报表主体

Project：横向工作任务，可关联一个或多个 Ledger，也可分配成员。
Register：业务模块内部台账，最终可生成或关联 AccountingEntry，但不是 Ledger。
```

## Team 与 Ledger 的边界

### Team 是协作范围

Team 用于回答：

- 谁和谁一起工作；
- 谁能看哪些账簿；
- 团队成员有哪些角色；
- 团队下有哪些客户或账簿。

Team 不应用于直接过滤凭证、报表、期初余额等正式财务数据。

### Ledger 是核算边界

Ledger 用于回答：

- 这张凭证属于哪个账簿；
- 这个期间属于哪个账簿；
- 这张报表使用哪个账簿的数据；
- 审计测试针对哪个账簿。

所有正式核算数据新增时，优先要求明确 `ledger_id`。

## Project 与 Ledger 的边界

Project 是工作任务边界，Ledger 是财务数据边界。

一个项目可以关联多个账簿，例如：

- 一个集团审计项目关联多个子公司账簿；
- 一个代理记账项目关联一个客户账簿；
- 一个税务专项项目关联一个或多个纳税主体及账簿。

规则：

1. 凭证、报表、期间、期初余额以 `ledger_id` 为主边界。
2. 审计计划、任务分配、负责人、项目状态以 `project_id` 为主边界。
3. 审计发现可同时关联 `project_id` 和 `ledger_id`。
4. 不应因为项目存在就省略账簿归属。

## Register 与 Ledger 的边界

Register 是业务台账，不等于会计账簿。

例如：

- 银行日记账；
- 发票台账；
- 固定资产卡片；
- 库存流水；
- 合同台账。

这些业务台账可以生成或关联会计凭证，但不能直接替代 Ledger。

规则：

1. Register 记录业务事实。
2. AccountingEntry 记录会计确认结果。
3. Register 到 AccountingEntry 之间需要规则、人工确认或 AI 草稿复核。
4. Register 模块不得绕过账簿权限直接写正式凭证。

### Register 与 Tag / 向量层（v0.3）

业务台账与分录、Tag 的关系见 [tag-vs-account-hierarchy.md §1.1–§1.2](../../backend/docs/tag-vs-account-hierarchy.md)。

1. **Register 只存模块核心登记字段**（如资产卡片号、发票号码、合同编号）；部门、项目、资产类别等**共用 EntryTag**，作为台账的辅助语义层，避免每模块重复建表。
2. **Tag / 向量体系对全平台开放**（固定资产、税务、进销存、合同、银行等），`ledger_id` 下共享 `TagCategory` 与向量 collection。
3. Register 可与 **分录 + Tag 双向映射**（从序时簿派生卡片、从卡片生成折旧凭证等），但**金额、科目、借贷仍以 `AccountingEntry` 为准**（§1.1 底线）。
4. 新模块禁止另起炉灶做平行标签字典或平行向量库，除非 §1.2 检查清单书面例外。

### Register 子类型边界（v0.2）

功能模块台账（Register）按**业务事实类型**拆分，禁止混登：

| Register | 登记对象 | 典型状态 | 不得登记 |
|---|---|---|---|
| `contract_register` 合同台账 | 签约承诺与条款 | 待执行 / 未执行 / 取消 / 已完成 | 应收应付余额、银行收付 |
| `counterparty_ledger` 往来款项台账 | 已发生的债权债务余额 | 应收 / 应付 / 预收 / 预付 | 纯合同承诺、银行流水明细 |
| `bank_cash_flow` 资金收付台账 | 银行账户实收实付 | 收入 / 支出 / 未达账项 | 往来余额（仅勾稽，不重复落库） |

链路：`合同(承诺) → 业务单证 → 往来余额 → 银行收付`；各段登记到对应 Register，通过勾稽关联而非重复写入。

详见 `.trae/documents/register-audit-workflow-plan.md`。

## Accounting Entity 与 Ledger 的边界

Accounting Entity 是会计主体或报表主体。

Ledger 是系统里的账簿数据边界。两者常常一一对应，但不强制等同。

可能场景：

1. 一个 Ledger 对应一个 Accounting Entity：普通单体企业。
2. 一个 Ledger 内包含多个内部管理主体：事业部、门店、部门。
3. 多个 Ledger 汇总到一个合并报表主体：集团合并。
4. 一个 Legal Entity 下存在多个管理账簿：多业务线独立核算。

规则：

- 普通凭证和报表先以 Ledger 为主。
- 涉及合并、分部、内部核算时，再引入 Accounting Entity。
- Accounting Entity 不应被 Team 或 Project 随意替代。

## Counterparty 的边界

Counterparty 是交易对象，不是科目，也不绑定借贷方向。

例如：

- 客户；
- 供应商；
- 关联方；
- 税务机关；
- 员工；
- 个人；
- 内部单位。

规则：

1. 主科目承担会计准则分类。
2. Counterparty 表达交易对象语义。
3. 借贷方向变化不改变 Counterparty 的身份。
4. 应收/预收、应付/预付重分类应看余额方向和列报规则，不应靠 Counterparty 改名。
5. 税费行不应硬编码“税务局”，除非确实需要政府类 Counterparty。

## EntryTag 与 DocumentTag 的边界

### EntryTag

EntryTag 绑定分录，用于表达分录语义，例如：

- 部门；
- 项目；
- 人员；
- 产品；
- SKU；
- 渠道；
- 区域；
- 客户/供应商语义；
- 业务线；
- AI 或人工补充的辅助核算标签。

EntryTag 可服务：

- 明细查询；
- 辅助核算；
- 向量检索；
- 审计风险识别。

但 EntryTag 不替代：

- 会计科目；
- 借贷方向；
- 正式凭证审核；
- 报表列报规则。

### DocumentTag

DocumentTag 绑定原始资料，用于表达文件证据语义，例如：

- 发票；
- 合同；
- 银行流水；
- 入库单；
- 结算单；
- 审计证据；
- 企业登记信息；
- 关联方线索。

DocumentTag 可服务：

- 原始资料分类；
- AI 证据检索；
- 风险线索；
- 凭证草稿生成。

但 DocumentTag 不直接构成正式凭证。

## AI 与正式财务规则的边界

AI 可以做：

1. 识别原始资料类型；
2. 提取摘要、金额、日期、对方单位；
3. 推荐科目和分录草稿；
4. 判断证据是否充分；
5. 生成 EntryTag 或 DocumentTag 候选；
6. 提示风险和补充资料；
7. 辅助解释系统操作。

AI 不可以直接做：

1. 绕过人工复核生成正式凭证；
2. 绕过账簿权限读取或写入数据；
3. 绕过结账规则修改已结账期间；
4. 代替确定性报表规则；
5. 代替审计人员最终判断；
6. 在证据不足时强行确认银行存款、收入、成本或费用。

## organization_id 的处理口径

当前项目早期存在 `organization_id` 字段或概念。

治理口径：

1. 新需求不再以 `organization_id` 作为正式财务数据主边界。
2. 凭证、分录、期初、期间、报表、审计测试优先使用 `ledger_id`。
3. 如果旧表或旧 API 仍保留 `organization_id`，应明确其兼容性质。
4. 后续迁移时，应评估是否将 `organization_id` 映射到 Team、Legal Entity 或 Accounting Entity，而不是默认等同于 Ledger。

## 典型边界案例

### 案例 1：导航排序

归属域：D03 Shell、导航、工作台、模块入口。

In Scope：

- MainShell 顺序；
- 一级菜单；
- 高亮；
- 父级默认跳转。

Out of Scope：

- 登录异常；
- dashboard API；
- 凭证生成；
- 审计导入；
- 数据库模型。

### 案例 2：序时簿导入

归属域：D06 审计证据与审计流程。

In Scope：

- `source_type=audit_day_book`；
- 凭证号分组；
- 借贷平衡检查；
- 跳号检测；
- 审计 Step3 报告展示。

Out of Scope：

- MainShell 导航；
- dashboard 首页；
- 会计科目模板；
- 正式总账凭证审核。

### 案例 3：AI 生成凭证

归属域：D04 凭证生命周期，依赖 D09 AI 草稿与 EntryTag。

In Scope：

- 原始资料到草稿；
- 证据充分性；
- 人工复核；
- 草稿转正式凭证；
- 留痕。

Out of Scope：

- 自动创建完整科目体系；
- 绕过复核直接入账；
- 改导航；
- 改审计报告导出。

### 案例 4：会计科目模板

归属域：D07 基础资料。

In Scope：

- 空白建账提示；
- 手工新增科目；
- 行业模板可选预览导入；
- 冲突提示；
- 不覆盖已有科目。

Out of Scope：

- AI 自动正式入账；
- 凭证 Step3 算法；
- 报表计算规则本身。

## 后续执行规则

1. 新需求先选一个主需求域。
2. 如果需求跨域，先拆分为多个 spec。
3. 一个 spec 不允许同时承担“导航 + 后端接口 + AI + 报表 + 审计”这种多域任务。
4. 缺陷修复只修缺陷，不能顺手扩展产品功能。
5. 所有正式财务数据优先明确 `ledger_id`。
6. 所有 AI 生成结果必须区分“建议 / 草稿 / 正式”。
7. 所有 checklist 完成时必须说明完成到 L0-L6 的哪一级。
