# Tasks

## Task 1：识别代码库和 UI 入口
- [x] 1.1 识别前端技术栈、路由结构和全局 Shell
- [x] 1.2 识别登录注册、首次引导、工作台、团队/账套/项目页面
- [x] 1.3 识别财务总账、凭证流程、基础资料、报表页面
- [x] 1.4 识别审计、银行、税务、固定资产、进销存、Agent 页面
- [x] 1.5 区分真实功能页面、占位页面和半成品页面

完成摘要：前端为 Vite + React 18 + TypeScript + React Router + Ant Design。路由集中在 `frontend/src/App.tsx`，全局 Shell 为 `frontend/src/layout/MainShell.tsx`。系统内页统一走 MainShell，登录/注册为公开页，首次引导也挂在 Shell 内。真实功能较完整页面包括登录、注册、首次引导、总工作台、团队管理、账套管理、项目生命周期、会计科目、往来单位、组织/人员、期初余额、期间、凭证人工录入/AI 生成、审计导入/测试/发现/导出、三张报表、Agent 助手。占位页面主要通过 `PlaceholderModulePage` 承载账簿、总账、明细账、银行账户/日记账/对账、发票、涉税助手、固定资产、进销存等预留入口。半成品页面主要是各模块工作台及部分管理页：有导航和样例状态，但部分统计/列表仍是静态示例或提示待实现。

## Task 2：评审全局视觉一致性
- [x] 2.1 检查 MainShell 的 Header、Sider、Content 布局一致性
- [x] 2.2 检查页面标题、卡片、表格、按钮、间距是否统一
- [x] 2.3 检查图标颜色和模块色彩是否专业且不过度杂乱
- [x] 2.4 检查登录、引导页与系统内页风格是否匹配
- [x] 2.5 标记视觉不一致或美观度较弱页面

完成摘要：MainShell 布局基础一致，Header 深色、Sider 浅色、Content 白底卡片化，整体偏企业后台风格。页面标题、卡片、表格和按钮总体使用 Ant Design 原生规范，视觉风险较低。主要不一致点：Shell 的 Content 已有 16px padding，但凭证/审计 Step 页面内部再次使用 24px padding 和 maxWidth，导致与普通管理页空间感不同；凭证 Step 中 Step 文案存在“选择模式/选择类型”“导出账套/生成账套”不一致；登录注册页与系统内页均为简洁白卡片风格，但品牌标题、说明层级较弱；工作台模块图标颜色较多，尚未形成稳定的模块色板；占位页过于朴素，容易让用户误以为功能缺失或页面未完成。

## Task 3：评审功能与流程体验
- [x] 3.1 检查工作台是否能清楚表达当前系统状态和下一步动作
- [x] 3.2 检查凭证 Step1-Step5 是否连贯、命名一致、操作可理解
- [x] 3.3 检查审计 Step1-Step6 是否连贯、命名一致、操作可理解
- [x] 3.4 检查基础资料页面是否适合财务人员维护档案
- [x] 3.5 检查错误提示、加载状态、空状态是否清晰

完成摘要：总工作台能展示待处理凭证、未结账期间、审计测试、账套缺失等状态，也有进入各模块按钮，适合作为首页。凭证流程主线可理解，尤其人工录入路径包含期间、凭证字/号、日期、科目、往来单位、借贷平衡校验，贴近会计录凭证习惯；但 Step3 已“确认并落库”后 Step4 又复核，业务顺序语义偏冲突，建议后续单独梳理。审计流程从范围、证据、序时簿、测试、发现、报告基本连贯，序时簿检测报告对审计人员有价值；但 Step1 的科目/年度选项为静态示例，Step2 与 Step3 都有上传概念，入口职责需要进一步说明。基础资料页面整体适合财务档案维护，会计科目和期初余额的业务语义较好；往来单位页仍偏基础 CRUD，关联方、统一信用代码等字段有价值但提示不足。错误提示和加载状态多数存在，空状态基本清晰，但占位模块与静态工作台缺少“这是预留功能/下一步去哪里”的统一说明。

## Task 4：从财务用户视角评估体验
- [x] 4.1 评估会计人员录凭证、选科目、选期间、选对方单位的易用性
- [x] 4.2 评估审计人员导入序时簿、查看风险、导出报告的易用性
- [x] 4.3 评估管理员建团队、建账套、建项目、首次引导的易用性
- [x] 4.4 标记“技术味太重”或财务语义不足的位置

完成摘要：会计人员路径中，人工凭证录入最贴近实务，可选会计期间、凭证字/号、科目和往来单位，并有借贷平衡提示；需优化处是“open/reopened”“jobId”“metadata”“JSON”等技术口径不应暴露给业务用户。审计人员路径中，序时簿导入、跳号/不平衡凭证、风险发现复核和报告导出较符合审计工作习惯；需增强审计范围与项目的真实关联。管理员路径中，首次引导对团队、账套、项目、会计主体的财务含义解释较好；团队/账套/项目管理可用，但团队成员仍显示用户 ID、账套授权显示团队 ID/角色英文，财务/管理语义不足。技术味较重位置集中在 Agent 页面“轻量模型/规则兜底/置信度”、税务助手“知识库 DAG”、凭证导出 JSON、部分页面展示 backend status 英文值。
