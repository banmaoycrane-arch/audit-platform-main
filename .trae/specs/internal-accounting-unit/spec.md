# 内部核算单位设计 - 产品需求文档

## Overview
- **Summary**: 设计支持多维度、可组合的内部核算单位系统，实现颗粒度原子化管理，支持跨主体核算和动态合并/分离考核场景，特别支持生产型企业的物料层级管理
- **Purpose**: 解决项目跨主体归集、考核维度变化等复杂核算需求，将辅助核算扁平化而非取消，支持行业差异化颗粒度管理
- **Target Users**: 财务人员、审计师、经营管理人员

## Goals
- 支持多维度核算单位（项目、部门、产品、渠道、客户、平台、职员等）
- 实现核算单位的颗粒度原子化（支持SKU级核算）
- 支持跨主体核算（一个项目跨越多个会计主体）
- 支持动态合并/分离考核场景
- 建立核算单位之间的层级关系和组合关系
- 支持物料层级管理（原材料→半成品→成品→商品SKU）
- 根据行业推荐最小颗粒度划分

## Non-Goals (Out of Scope)
- 不涉及具体的成本分配算法
- 不处理预算编制逻辑
- 不实现绩效考核系统

## Background & Context
内部核算单位是企业进行经营管理和绩效考核的基础，需要考虑：
1. **跨主体核算**：A项目可能跨越多个会计主体归集费用、收入、成本
2. **动态考核**：项目后期可能存在合并、分离考核的场景
3. **颗粒度问题**：辅助核算不是取消而是扁平化，将核算单位颗粒度原子化
4. **最小颗粒度**：职员、部门、店铺、产品线、业务单位、产品、SKU、渠道、大客户事业部、平台等
5. **物料层级**（生产型企业）：原材料→半成品→成品→商品SKU
6. **行业差异**：不同行业有不同的核算颗粒度需求

### 常见内部核算颗粒度
| 颗粒度类型 | 说明 | 适用场景 |
|-----------|------|---------|
| 职员 | 最小人力单位，用于人员成本归集 | 薪酬核算、绩效考核 |
| 部门 | 组织架构单位 | 部门预算、费用管控 |
| 产品 | 具体产品或服务 | 产品利润分析、成本核算 |
| SKU | 库存管理最小单位（Stock Keeping Unit） | 库存管理、成本追踪 |
| 渠道 | 销售渠道或平台 | 渠道业绩分析 |
| 客户 | 具体客户或客户群体 | 客户价值分析、应收账款 |
| 项目 | 一次性业务活动 | 项目成本核算、效益评估 |
| 店铺 | 线下或线上门店 | 门店经营分析 |

### 生产型企业物料层级关系
生产型企业的物料流转遵循严格的层级关系：

```
原材料 → 半成品 → 成品 → 商品SKU
   │         │       │        │
   ▼         ▼       ▼        ▼
  提炼/    装配/    规格化   组合包装
  加工    化学发酵   差异化   渠道定制
```

**详细说明：**
- **原材料**：未经加工的原始物料，如矿石、棉花、化工原料
- **半成品**：经过初步加工但尚未完成的中间产品
  - 原材料通过组装或提炼转化为半成品
  - 半成品可作为其他产品的原料
- **成品**：完成全部生产流程的最终产品
  - 半成品通过装配或化学发酵转化为成品
  - 成品具备完整使用功能
- **商品SKU**：面向市场销售的最终单元
  - 成品取不同规格形成不同SKU
  - 多个成品组合为一个商品SKU
  - 不同包装规格形成不同SKU

**BOM（物料清单）关系示例：**
```
商品SKU "iPhone 15 Pro 256GB 蓝色"
└── 成品 "iPhone 15 Pro 主机"
    ├── 半成品 "A17 Pro芯片"
    │   ├── 原材料 "硅晶圆"
    │   └── 原材料 "金属触点"
    ├── 半成品 "显示屏组件"
    │   ├── 原材料 "OLED面板"
    │   └── 原材料 "触控玻璃"
    └── 半成品 "外壳组件"
        └── 原材料 "钛金属"
```

### 行业推荐颗粒度划分
| 行业类型 | 推荐最小颗粒度 | 说明 |
|---------|-------------|------|
| **生产制造** | SKU/成品 | 支持原材料→半成品→成品→SKU全流程核算，精细追踪物料成本 |
| **化工行业** | 批次/配方 | 按生产批次和配方核算成本，追踪原料配比 |
| **电子制造** | PCB板/元器件 | 按电路板型号和元器件级别核算 |
| **食品加工** | 批次/配方 | 按生产批次和配方追溯 |
| **贸易批发** | 商品/批次 | 按商品类别或批次核算库存和成本 |
| **零售电商** | SKU/店铺 | 按SKU和店铺维度核算销售业绩 |
| **连锁零售** | 门店/SKU | 按门店和SKU进行精细化管理 |
| **服务业** | 项目/服务项 | 按项目或服务类型核算收入和成本 |
| **软件服务** | 模块/许可证 | 按软件模块或许可证类型核算 |
| **金融** | 产品/账户 | 按金融产品或客户账户核算 |
| **银行** | 产品/客户 | 按金融产品和客户维度分析 |
| **保险** | 保单/险种 | 按保单和险种类型核算 |
| **物流运输** | 订单/线路 | 按运输订单和线路核算成本 |
| **建筑施工** | 项目/分部工程 | 按项目和分部工程核算成本 |
| **医药医疗** | 药品/批次 | 按药品和生产批次追踪 |
| **农业** | 作物/地块 | 按作物种类和地块核算 |

## Functional Requirements
- **FR-1**: 支持多类型核算单位定义（项目、部门、产品、渠道、客户、平台、职员等）
- **FR-2**: 实现核算单位的层级关系（如店铺→产品、部门→员工）
- **FR-3**: 支持核算单位的组合关系（如项目=店铺A+店铺B）
- **FR-4**: 支持跨主体核算（一个核算单位关联多个会计主体）
- **FR-5**: 支持核算单位的动态合并/分离
- **FR-6**: 提供核算单位语义检索与关联分析
- **FR-7**: 记录核算单位的历史版本变化
- **FR-8**: 支持物料层级管理（原材料→半成品→成品→商品SKU）
- **FR-9**: 根据行业推荐核算颗粒度
- **FR-10**: 支持物料BOM（物料清单）管理

## Non-Functional Requirements
- **NFR-1**: 支持至少10000个核算单位的管理
- **NFR-2**: 核算单位组合查询响应时间 < 500ms
- **NFR-3**: 支持至少10层嵌套层级关系
- **NFR-4**: 支持至少100000个SKU的管理

## Constraints
- **Technical**: 基于现有数据库模型扩展
- **Business**: 需符合企业内部管理核算要求
- **Dependencies**: 依赖现有的会计主体系统

## Assumptions
- 用户理解内部核算单位的概念
- 核算单位类型可以预先定义
- 层级关系是有向无环图（DAG）
- 用户理解物料BOM的概念

## Acceptance Criteria

### AC-1: 多类型核算单位定义
- **Given**: 系统已配置核算单位类型
- **When**: 创建核算单位
- **Then**: 系统支持选择类型（项目、部门、产品、渠道等）
- **Verification**: `programmatic`

### AC-2: 层级关系管理
- **Given**: 存在父级核算单位
- **When**: 创建子级核算单位
- **Then**: 系统建立层级关系
- **Verification**: `programmatic`

### AC-3: 跨主体核算
- **Given**: 一个项目跨越多个会计主体
- **When**: 查询项目核算数据
- **Then**: 系统返回跨主体的汇总数据
- **Verification**: `programmatic`

### AC-4: 动态合并/分离
- **Given**: 存在多个核算单位
- **When**: 用户执行合并/分离操作
- **Then**: 系统创建新的组合或拆分现有组合
- **Verification**: `human-judgment`

### AC-5: 语义检索
- **Given**: 存在多个核算单位
- **When**: 输入模糊名称
- **Then**: 系统返回匹配的核算单位
- **Verification**: `programmatic`

### AC-6: 物料层级管理
- **Given**: 存在原材料、半成品、成品、SKU
- **When**: 创建物料关系
- **Then**: 系统建立物料层级（原材料→半成品→成品→SKU）
- **Verification**: `programmatic`

### AC-7: 行业颗粒度推荐
- **Given**: 用户选择行业类型
- **When**: 创建核算单位
- **Then**: 系统推荐合适的颗粒度级别
- **Verification**: `human-judgment`
